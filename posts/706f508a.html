07.项目质量管理

Pupper2024-07-142024-10-10
~启动规划执行监控收尾
质量管理~1.规划质量管理2.管理质量3.控制质量~

1. 项目质量管理概述

项目质量管理: 把组织的质量政策应用于规划、管理、控制 项目和产品 的质量要求, 以满足相关方目标的过程

  • 项目质量管理: 针对的是项目过程中 所涉及的活动, 适用于 所有领域 的项目, 于项目性质关系不大
  • 产品质量管理: 针对的是项目生成的 具体可交付成果,与可交付成果的 性质和特性 密切相关,测量方法和技术需要结合具体的 可交付成果类型 来确定

1.1 质量与等级对比:

  • 质量: 表示满足用户需求, 无明显缺陷
  • 等级: 表示技术或工艺的复杂程度

1.2 精确与准确对比:

  • 精确: 表示精密程度
  • 准确: 表示期望与实际的接近程度, 是对正确性的评估

1.3 现代质量管理原则

  • 质量是 符合要求 和 适合使用
  • 反对 镀金
  • 质量是 管理层 的责任, 85%
  • 预防 胜于检查
  • 规划 质量、管理 质量、控制 质量
  • 持续改进, PDCA 循环、六西格玛、TQM
  • 与供应商的互利合作, 着眼于 长期关系

1.4 五种质量管理水平

  1. 用户发现缺陷, 售后补救, 代价最大
  2. 控制质量过程, 检测和纠正缺陷, 售前补救, 代价较大
  3. 通过质量保证检查并纠正过程本身, 过程纠正, 代价较小
  4. 将质量融入项目和产品的规划和设计中,提前预防, 代价更小
  5. 在整个组织内创建一种关注并致力于实现过程和产品质量的文化, TQM

2. 规划质量管理

  • 规划质量管理: 识别项目的质量要求或标准, 并书面描述如何满足这些要求或标准的过程
    • 为在整个项目期间如何管理质量提供 指南和方向
  • 本过程仅开展一次或仅在项目的预定义点开展

2.1 工具和技术

  • 成本效益分析: 项目团队应该评估和选择 高效益、低成本 的做法,提高质量管理的成本效益
  • 👍 质量成本 的分类:
    • 预防成本: 预防特定项目质量低劣所带来的相关成本
      • 如: 培训、流程改进、质量计划
    • 评估成本: 评估、测量、审计和测试特定项目所带来的相关成本
      • 如: 检查、测试、审计、破坏性测试
    • 内部失败成本: 项目过程中发现的失败而造成的成本
      • 内部: 重做、修复、报废
    • 外部失败成本: 客户发现的失败而造成的成本
      • 外部: 保修、赔偿、诉讼
  • 多标准决策分析: 针对某个质量测量指标的优先级做出判断
  • 流程图: 也叫过程图,用于显示输入与输出之间的关系
    • 关键要素: 活动、决策点、分支循环、并行路径、整体处理顺序等
    • 有助于识别可能出现质量缺陷或纳入质量检查的地方
  • 逻辑数据模型: 把组织数据可视化,用商业语言描述, 不依赖于特定技术而评估质量的方法
  • 矩阵图: 在行列交叉的位置展示因素、原因和目标之间的强弱关系
    • 行表示因素、目标, 列表示原因, 交叉点表示强弱关系
  • 测试与检查的规划: 拟定执行和控制过程需要采用的测试和检查方法的技术

2.2 输出

  • 质量管理计划: 描述如何实施适用的政策、程序和指南, 以满足项目的质量要求
    • 包含: 采用的质量标准、质量目标、角色与职责、质量工具、持续改进程序等
  • 质量测量指标: 用于描述项目及控制质量过程中的验证指标
    • 如: 任务完成百分比、故障率、缺陷数量、客户满意度等

2. 管理质量

  • 管理质量: 将质量管理计划转化为可执行的质量活动的过程
    • 提高实现质量目标的可能性, 以及识别无效和导致质量低劣的原图
    • 使用控制质量过程数据和结果向相关方展示项目总体质量状态

2.1 工具和技术

  • 核对单: 一种结构化工具,用来核实需求列表是否已得到满足
  • 过程分析: 对工作开展过程进行分析的做法, 提高工作效率, 让价值和收益最大化
    • 通过过程分析, 寻找 不增值 的地方、发现问题、持续改进
  • 根本原因分析: 根本原因一般只有 1 个, 主要原因有多个
    • 分析根本原因 —> 查找问题 —> 解决问题 —> 验证结果
  • 因果图: 也叫 鱼骨图,将问题的原因分解为 离散分支, 以便更好的查找根本原因
  • 直方图: 也叫 柱状图,用于展示缺陷数量、类型、出现次数等
    • 配合 帕累托图 使用更易于表现问题的严重性
    • 帕累托法则: 20/80 原则, 20% 的问题来自于 80% 的原因
  • 散点图: 也叫 关联图, 用于展示两个变量之间的关系
    • 一个轴表示过程、环境、或活动的任何因素, 另一个轴表示质量缺陷
      散点图

  • 👍 审计: 用于确定项目活动是否遵循了组织、政策、过程和程序的一种 结构化 且 独立 的过程
    • 通常是由 项目外部 的团队开展, 如: 内部审计部门、PMO、外部审计公司
  • 面向 X 的设计 (DfX): 是产品设计期间采用的一系列技术指南,旨在优化设计的特定方面,可以控制或提高产品最终特性
    • X 可以是: 可靠性、可维护性、可用性、可支持性等
    • 使用 DfX 设计法则, 可以降低成本、改进质量、提供绩效和客户满意度
  • 👍 问题解决: 使用 结构化 的问题解决方法有助于消除问题和定制长久有效的解决方案
    • 问题解决流程: 识别问题 —> 定义问题 —> 识别根本原因 —> 生成备选方案 —> 选择最佳方案 —> 实施方案 —> 验证方案有效性
  • 质量改进方法: PDCA 和 六西格玛 是常用于分析和评估改进机会的两种质量改进工具
    • PDCA: 计划 —> 实施 —> 检查 —> 处理
    • 六西格玛: 定义问题 —> 测量问题 —> 分析问题 —> 改进问题 —> 控制问题

2.2 输出

  • 质量报告: 描述项目的质量状况, 可能以图形、数据或文件的形式呈现
    • 包含: 质量管理问题、改善建议、纠正措施、预防措施等
  • 测试与评估文件: 可基于行业需求和组织模板创建测试与评估文件, 作为 控制质量过程 的输入

3. 控制质量

  • 控制质量: 为了评估绩效, 确保项目 输出完整、正确且满足客户期望, 而监督和记录质量管理活动执行结果的过程
    • 核实项目可交付成果,工作已经达到主要相关方的质量要求, 可供最终验收
  • 本过程根据需要在整个项目期间
  • 目的在于测量产品的 完整性、合规性、适用性, 用 可靠的数据 来证明项目已经达成了质量目标

3.1 输入

  • 批准的变更请求: 需要确认批准的变更请求是否已经实施, 且准确无误
  • 可交付成果: 在某个过程或阶段,必须产出的可被检测或核实的产品

3.2 工具和技术

  • 核查表: 也叫 计数表, 用于记录缺陷、问题、错误、变更等的数量
  • 检查: 用于评估产品是否符合质量标准的一种技术
    • 传统项目: QC 团队通过检验产品, 来确定是否符合质量标准
    • 敏捷项目: 开发团队自组织自我管理,通过集体检查和不断测试,来验证工作产品是否符合质量标准
  • 测试/产品评估: 有组织的, 根据产品需求和规范, 对产品进行测试和评估
    • 目的在于找出产品的缺陷, 以便及时纠正
  • 控制图: 用于确定一个过程是否稳定, 是否具有可预测的绩效
    • 控制界限是根据标准的统计原则, 通过标出的统计计算确定,代表一个稳定过程的自然波动范围
    • 控制线应该是距离均值 3 倍标准差, 控制线应该是距离均值 4 倍标准差

控制图的 七点规则: 连续 7 个或 7 个以上的点落在均值的同一侧, 则说明过程失控, 需要查明原因并调整过程

3.3 输出

  • 质量控制测量结果: 是质量控制活动的书面记录结果
  • 核实的可交付成果: 确定可交付成果的正确性和完整性, 以便最终验收
  • 确认范围 表示需求有没有做, 而 控制质量 表示做的对不对
