1. 项目范围管理

  • 项目范围管理: 是确保项目包含了所有工作，且仅包含项目所有工作，以完成项目的目标。
  • 管理项目范围: 主要在于 定义和控制 项目的工作内容，确保项目的目标能够得以实现。
  • 范围管理是其他项目管理过程的基础，因为它定义了项目的工作内容，而其他过程则是在这个基础上进行的。

2. 规划范围管理

  • 规划范围管理: 是为记录如何定义、确认和控制项目及产品范围,从而创建范围管理计划的过程
  • 作用: 在整个项目期间如何管理范围提供指导, 有利于降低 范围蔓延 的风险
  • 本过程仅开展一次或仅在项目的预定义点开展

2.1 输出

  • 范围管理计划: 描述将如何定义、制定、监督、控制和确认项目范围的文件,是项目管理计划的组成部分
    • 包含: 范围确认、范围控制、范围基准等
    • 是 定义范围、创建 WBS、确认范围、控制范围 的输入
  • 需求管理计划: 描述如何分析、记录和管理需求的过程,是项目管理计划的组成部分
    • 包含: 需求的来源、分类、优先级、验收标准等

3. 收集需求

  • 收集需求: 是为实现目标而 确定、记录和管理 相关方的需要和需求的过程
  • 作用: 为定义 产品范围 和 项目范围 提供基础

3.1 工具和技术

  • 问卷调查: 指设计一系列问题, 向众多受访者 快速收集信息 的方法
    • 特点: 多样化、快速、地理分散等
  • 标杆对照: 指将项目实践或质量标准与其他可比项目比较, 以识别 最佳实践,为 绩效考核 提供依据
    • 特点: 有助于识别最佳实践, 但不一定适用于所有项目
  • 投票: 指通过 多数原则 或 权重投票 等方式, 以确定 优先级 或 选择 的方法
    • 特点: 有助于快速确定优先级, 但可能导致少数人的需求被忽略
  • 多标准决策分析: 指通过 多个标准 来评估和选择 多个选项 的方法
    • 特点: 有助于综合考虑多个因素, 但需要投入较多的时间和资源(如: 权重占比的形式)
  • 亲和图: 指通过 分组 的方式, 将 相关需求 之间的 关系 可视化的方法
    • 特点: 有助于识别需求之间的关系, 但可能导致需求的遗漏
  • 思维导图: 把头脑风暴中获取的创意整合成一张图, 以反映创意之间的 共性和差异,激发新的创意
  • 名义小组技术: 用于促进 头脑风暴 的一种技术,通过 投票 获取最有用的创意, 以确定 优先级 的方法
  • 观察和交谈: 直接查看个人在各自的环境中如何执行工作和实施流程.
    • 适用于: 客户难以表达需求, 需要进一步了解需求的情况
  • 引导: 通过 引导 的方式, 促使相关方 自行发现 和 表达 需求
    • 适用于: 相关方难以表达需求, 需要引导相关方表达需求
  • 系统交互图: 用于描述 系统 的 功能 和 交互 的图形表示法
    • 适用于: 有助于理解系统的功能和交互, 但可能过于复杂
  • 原型法: 在实际制造之前, 制作一个 模型 或 原型 来展示产品的功能和特性
    • 适用于: 有助于理解产品或系统的功能和特性, 但可能过于复杂

3.2 输出

  • 需求文件: 记录各种单一需求将如何满足与项目相关的业务需求,是逐渐细化的过程
    • 是 定义范围、创建 WBS、确认范围、控制范围 的输入
  • 跟踪矩阵: 用于跟踪需求的来源, 以及需求如何满足项目目标的矩阵

4. 定义范围

  • 定义范围: 制定项目和产品详细描述的过程, 以满足相关方的需求和期望
  • 作用: 明确收集的需求那些需要包含在项目内, 从而描述产品、服务获成功的 边界 和 验收标准.
  • 反复开展, 逐步明细化

4.1 工具和技术

  • 产品分析: 把高层级的产品或服务转变为有意义的 可交付成果 (渐进明细化)
    • 产品分析的技术包含: 产品分解、系统工程、需求分析、系统分析、价值工程等

4.2 输出

  • 范围说明书: 详细描述项目和产品的特征和功能, 以满足相关方的需求和期望
    • 是 定义范围、创建 WBS、确认范围、控制范围 的输入
    • 范围说明书 更加 详细、具体, 而 项目章程 则是 高层级 的描述`
项目章程与范围说明书

5. 创建工作分解结构(WBS)

  • 工作分解结构: 把项目可交付成果和项目工作分解成更小、更易于管理的组件
    • WBS 只包含项目范围内的工作, 不包含项目范围以外的工作(只做该做的)
    • 将复杂的问题 简单化, 繁杂的工作分解细化
    • WBS 更关注成果本身, 而非过程
  • 作用: 为所要交付的内容提供架构
  • 仅开展一次或仅在项目的预定义点开展

WBS 分解原则:

  1. 100% 原则: 即没有遗漏
  2. 包含原则: 各个层级之间有包含关系
  3. 不可再分原则: 即分解需要进行到底
  4. 信息透明原则: 团队参与
  5. 80 小时原则: 以 2 周时间为节点,超出表示还可以再分解
  6. 独立负责: 工作包需要有负责人
  7. 滚动式规划原则: 渐进明细
  8. 4-6 层原则: 分解层级限制
WBS 表现形式

以 树状图 的形式更容易表现出来

WBS 中的工作包、规划包、控制账户的关系

  • 工作包: 带有独特标识号的 WBS 最底层工作
  • 控制账户: 一个包含两个以上的管控点,把范围、预算和进度加以整合并与挣值比较,以测量绩效
  • 规划包: 详细进度位置无法拆解的工作包

5.1 输出

  • 范围基准: 经过批准的 范围说明书、WBS 和相应的 WBS 词典, 只有通过正式的变更控制程序才能进行变更
    • 范围基准是项目管理计划的组成部分
  • WBS 词典: 针对 WBS 中的每个组件,详细描述可交付成果、活动和进度信息的文件
    • 作用: 对 WBS 提供支持,其大部分信息由其他过程创建, 并在后期添加到词典中
    • 主要包括: 工作描述、负责组织、进度里程碑、所需资源等等

6. 确认范围

  • 确认范围: 是正式验收已完成的项目、成果的过程
  • 作用: 使验收过程具有客观性, 通过确认每个可交付成果来提高最终产品验收的可能性

6.1 输入

  • 核实的可交付成果: 已经完成并被控制质量过程检查通过的可交付成果

6.2 输出

  • 验收的可交付成果: 项目团队已经确认并获得相关方的认可签字的可交付成果

7. 控制范围

  • 控制范围: 监督项目和产品的范围状态, 管理范围基准变更的过程
  • 作用: 在整个项目期间对范围基准进行把控和维护

范围蔓延、镀金、渐进明细的比较:

  • 范围蔓延: 未经控制, 项目或产品范围扩大
  • 镀金: 团队主动增加超出范围的额外工作
  • 渐进明细: 随着信息越来越多, 估算越来越准确, 从而提高项目管理计划详细度

7.1 工具与技术

偏差分析: 确认实际绩效与基准的差异程度及原因的一种技术

趋势分析: 通过比较项目绩效数据的变化趋势, 以确定项目绩效是否在变好或变坏
